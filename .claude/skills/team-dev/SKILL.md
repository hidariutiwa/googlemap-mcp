# Team Development Skill

ユーザーの機能要望 `$ARGUMENTS` に対して、Agent Teamを編成し並列で実装を行う。

## Phase 1: 要望分析

1. `$ARGUMENTS` を機能要望として解析する
2. コードベースを探索し、以下の既存パターンを把握する:
   - `src/googlemap_mcp/tools/places.py` — Pydantic BaseModel + httpxクライアントパターン
   - `src/googlemap_mcp/main.py` — FastMCPインスタンスへのツール登録パターン
3. 実装計画を立案する:
   - 新規作成が必要なファイル
   - 変更が必要な既存ファイル
   - 必要なテスト

## Phase 2: チーム作成 & タスク割り当て

以下の2人のチームメイトを生成する:

### implementer（実装担当）
- 本番コードの実装を担当
- クライアントクラス、Pydanticモデル、MCPツール登録を作成
- `src/googlemap_mcp/tools/` 配下の既存パターンに厳密に従うこと
- 作業対象: `src/` 配下のファイルのみ

生成時のプロンプト例:
```
あなたは implementer（実装担当）です。
以下の機能を実装してください: [機能の説明]

既存パターンに従ってください:
- src/googlemap_mcp/tools/places.py のクライアントクラスパターン（Pydantic BaseModel、httpx、PrivateAttr）
- src/googlemap_mcp/main.py のFastMCPツール登録パターン

作業手順:
1. 既存コードを読んでパターンを把握
2. 新しいクライアントクラスを作成
3. main.py にMCPツールとして登録
4. 完了したらリーダーにメッセージで報告

注意: テストは書かない。テストは tester が担当する。
```

### tester（テスト担当）
- テストコードの作成と実行を担当
- pytestを使用
- エッジケースも含めたテストを作成
- 作業対象: `tests/` 配下のファイルのみ

生成時のプロンプト例:
```
あなたは tester（テスト担当）です。
以下の機能のテストを作成してください: [機能の説明]

作業手順:
1. implementer の実装コードを読んで仕様を理解
2. tests/ ディレクトリにテストファイルを作成
3. 正常系・異常系・エッジケースのテストを網羅
4. python -m pytest でテストを実行
5. 結果をリーダーにメッセージで報告

注意: 本番コードは変更しない。テストのみを担当する。
```

### タスク作成の手順

1. 機能の内容に応じてタスクを作成する
2. 典型的なタスク構成:
   - タスク1: 「[機能名]クライアントクラスの実装」→ implementer に割り当て
   - タスク2: 「main.py へのMCPツール登録」→ implementer に割り当て（タスク1に依存）
   - タスク3: 「ユニットテストの作成」→ tester に割り当て（タスク1に依存）
   - タスク4: 「テスト実行と検証」→ tester に割り当て（タスク2, 3に依存）
   - タスク5: 「最終レビューと統合確認」→ リーダー自身が担当（タスク4に依存）

## Phase 3: 並列作業の開始

- 各チームメイトを生成してタスクを開始させる
- tmux split-pane でそれぞれのペインに表示される
- ファイル競合を避けるため、各エージェントの作業領域を明確に分離する:
  - implementer → `src/` 配下のみ
  - tester → `tests/` 配下のみ

## Phase 4: 監視・調整

- チームメイトからのメッセージを受信して進捗を監視する
- 問題が発生した場合:
  - 追加タスクを作成して割り当てる
  - メッセージで具体的な修正指示を送る
- 全タスク完了後、リーダー自身が最終レビューを実施:
  - 新規・変更ファイルを読んでコード品質を確認
  - テストをもう一度実行して全パス確認

## Phase 5: 完了報告

- チームメイトをシャットダウンする
- ユーザーに以下を報告する:
  - 実装した機能の概要
  - 作成・変更したファイル一覧
  - テスト結果（パス数、カバレッジ）
  - 既知の制限事項や今後の改善点（あれば）
